# Bug 复现验证闭环

当用户的反馈证明前一次验证过浅时，使用这个 skill。目标不是继续解释，而是进入可复现、可修复、可验证的闭环，并尽量覆盖用户实际触发问题的同一条路径。

## 触发信号

出现以下任一情况时，应升级为缺陷闭环：

- 用户反馈一个已声称修复的问题仍然失败。
- 用户说 UI 元素、桌面窗口、进程、服务或集成突然消失、退出或状态异常。
- 用户指出“应该自动验证”“你没有测到真实路径”。
- 用户观察到的现象与命令输出、静态代码阅读或先前判断矛盾。
- 功能依赖事件回调、GUI 自动化、后台进程、定时器、hook、IPC、文件监听、提示音、启动脚本或隐藏进程。
- 修复方案准备加入兜底、重试、延长超时、sleep、降级、默认值、吞异常或宽泛 catch，但还没有证明失败是如何产生的。

一旦触发，不要停在“建议修复”。只要可行，就要构建或运行一个能在修复前失败、修复后通过的验证闭环。

## 必须执行的流程

1. 先确认现场状态。
   - 检查进程、服务、窗口、状态文件、IPC 或状态桥是否真的存活。
   - 用命令、日志、截图、UI 自动化或状态文件捕获当前证据。
   - 如果用户说 UI 消失，先验证进程是否仍在，而不是假设只是隐藏或最小化。

2. 复现用户的真实路径。
   - 优先使用用户同一层级的交互：点击菜单、按快捷键、触发 hook、运行已安装命令、重启服务、打开真实 UI。
   - 不要用更低层替代路径冒充验证充分。例如 CLI 状态更新成功，不能证明 WPF 菜单点击安全。

3. 静默失败时先加观测能力。
   - 在事件处理、后台循环、异常边界、进程启动处添加本地日志。
   - 对默认会静默退出的运行时补未处理异常捕获。
   - 必要时在 status 输出中暴露日志路径。

4. 先证明问题如何产生。
   - 区分症状和原因。超时、卡死、空结果、保存失败、UI 不响应、偶发失败只是观察到的症状，不是原因。
   - 原因必须解释产生机制，例如文件锁、阻塞 UI 线程、死锁、错误状态迁移、竞态、资源泄漏、设备断开、网络不可用、权限拒绝、错误完成检测。
   - 用日志、异常类型、时序、trace、返回码、复现步骤、文档或受控实验支撑原因判断。
   - 证据不足时，先补诊断，不要用方便的表象归因推进修复。

5. 审查兜底是否正当。
   - 允许为正确系统中仍必然存在的异常写兜底，例如网络断开、远端服务不可用、电源中断、文件被其他进程锁定、用户取消、设备断开、权限拒绝。
   - 禁止用兜底补偿未诊断的问题。不要因为导出或保存偶发失败就直接延长超时；不要用 sleep 修顺序问题；不要用宽泛 catch、默认值或静默重试隐藏失败。
   - 写兜底前必须明确失败条件、异常分类、证据、重试/超时/取消/上报契约、可能隐藏的正确性风险、验证方法。
   - 如果失败属于流程不完备、状态不清、同步条件不明、API 用法错误或观测能力不足，先修主流程或补诊断。

6. 窄修根因。
   - 修失败路径本身，不只修表面现象。
   - 如果事件回调、定时器或后台进程可能导致应用退出，要在该边界隔离异常。
   - 保留用户已有改动，避免无关重构。

7. 创建自动验证产物。
   - 如果该问题类型未来可能回归，应添加仓库内验证脚本。
   - 验证应尽量覆盖真实失败路径，而不只是 parse/build 测试。
   - 断言进程存活、状态符合预期、日志没有本次验证期间的新错误、外部可见行为正确。
   - 对兜底逻辑要验证触发条件、重试预算、超时、取消、错误上报和不会隐藏数据损坏或状态不一致。

8. 端到端运行验证。
   - 按用户实际使用方式安装、重启、reload 或运行。
   - 如果命令层路径和真实交互路径不同，两者都要测。
   - 最终补丁完成后再次运行验证，而不是只在中途运行。

9. 汇报证据，而不是汇报信心。
   - 说明根因、实际运行的验证命令、当前 live 状态和剩余限制。
   - 如果增加了兜底，说明它处理的是必然异常还是流程缺陷；流程缺陷不应以兜底作为最终修复。
   - 如果没有 remote、不能 push、不能部署或不能访问真实环境，要明确说明。

## 验证清单

UI 或桌面应用至少考虑：

- 脚本或构建解析通过。
- 应用或进程能启动并持续存活。
- 真实菜单、按钮、快捷键或窗口路径可用。
- 后台 timer、watcher、hook tick 不会让应用退出。
- 状态文件、IPC 或状态桥按预期更新。
- 本次验证期间日志没有新的未处理异常、菜单动作失败或后台循环错误。
- 没有把 UI 卡死、超时、空结果、偶发失败等症状直接当成原因。

服务或 CLI 集成至少考虑：

- 验证已安装命令或 shim，而不只验证源码脚本。
- 验证 restart、reload 或重新安装后的行为。
- 验证状态能在实际运行时持久化。
- 进程退出时能留下可定位的失败日志。
- 如果存在重试、超时、降级或默认值路径，验证其触发条件、边界和错误上报。

## 完成标准

不要在以下条件满足前宣称修复完成：

- 已直接覆盖用户报告的失败路径，或明确说明无法覆盖的原因。
- 修复已经安装或应用到用户实际使用的运行时。
- 对可回归的问题类型建立了自动验证。
- 最终补丁后重新运行过验证。
- 当前状态来自 live evidence，而不是推测。
- 已说明症状和原因机制的区别；没有用兜底掩盖未查明根因。

如果验证被阻塞，继续收集最强的非破坏性证据，并明确说明缺失的检查项。
