# UI 路线学习到自动化脚本

## 核心判断

把未知 UI 自动化分成三个阶段：学习、固化、推广。学习阶段允许摸索，但必须留下路线状态和证据；固化阶段只把已经小步验证的动作写入脚本；推广阶段必须用干净项目或独立样例证明脚本可复现。

这个 skill 要和 `route-governance` 一起使用。先用 route 文件锁定当前路线，再做 UI 操作。不要靠聊天记忆判断“刚才哪一步是对的”。

## 工作流

1. **建立任务根目录**
   - 创建一个 disposable task root，用来放 route-state、截图、UIA 树、日志、脚本输出和最终文档。
   - 不要直接在用户真实项目上摸索，除非用户明确要求且已有备份。

2. **从知识库和用户提示提取候选路线**
   - 把知识库材料、软件帮助、历史脚本、用户口述步骤拆成可验证动作。
   - 用户提示的优先级高于模型猜测；如果用户指出“这里不是通信设置，而是项目树单元配置”，立即把错误路线记录为废弃路线。
   - 对每个候选动作写清楚：入口、前置状态、动作表面、期望观察、失败边界。

3. **用 route-governance 锁定当前路线**
   - 初始化或读取 `<task-root>/route-state.json`。
   - 为当前路线声明允许的 action surface，例如 `uia_observation`、`keyboard_shortcut`、`win32_window_probe`、`mouse_click`。
   - 禁止没有登记的 fallback。比如键盘路线失败后，不能悄悄改成坐标点击并声称路线成功。

4. **小步验证，不急着连续脚本化**
   - 每次只验证一个状态转换：打开窗口、展开树、选择列表、进入子对话框、写入字段、点 OK、触发编译。
   - 每一步都保存独立证据：UIA 节点、窗口标题、截图、结果 JSON、编译文本、日志。
   - 一个动作成功只证明该动作成功，不证明整条流程稳定。

5. **把成功片段先做成可重复脚本**
   - 当某个片段已经连续可重复，先脚本化该片段，用它快速创造下一个突破环境。
   - 不要等整条路线完全懂了才写脚本；也不要把未验证的后续步骤塞进脚本。
   - 脚本输出必须包含：输入参数、实际选中的 UI 节点、窗口状态、错误码、证据路径。

6. **失败分类和反思**
   - 两次相似失败后停下来反思，不要增加 sleep 或扩大点击区域来掩盖原因。
   - 常见错误族：入口错误、焦点错误、模式错误、观察对象错误、动作表面漂移、弹窗未处理、验证 oracle 错误。
   - 如果用户纠正了路线，把纠正写入 route 文件和文档，再继续。

7. **形成稳定连续脚本**
   - 只有当每个片段都有同轮证据，并且连续脚本在干净项目上通过，才能称为稳定。
   - 稳定脚本应当拒绝未验证能力，而不是假装支持。例：自动注册 ESI 未稳定时，参数可以保留但必须报明确错误码。
   - 稳定脚本应避免覆盖用户其他脚本或真实项目；默认输出到 task root 或显式 OutDir。

8. **更新 skill 或项目文档**
   - 文档只写已验证路线、证据路径、边界和失败条件。
   - 不要把探索中的猜测写成结论。
   - 修改 skill 后要提交；若用户要求，push。

## 路线状态模板

使用 `route-governance` 的 `route_guard.py`，至少记录这些内容：

```text
route_id: ui_<software>_<operation>_v1
owner: user_specified | agent_selected | fallback
commitment_level: locked | preferred | exploratory
preconditions:
  - 软件已打开到可编辑模式
  - 当前项目为 disposable 项目
  - 没有阻塞弹窗
action_surface.allowed:
  - uia_observation
  - keyboard_shortcut
  - win32_window_probe
action_surface.forbidden:
  - coordinate_click
  - unrecorded_fallback
observation_surface:
  - 目标窗口标题
  - 目标控件名称或 AutomationId
  - 输出 JSON/截图/编译文本
failure_boundary:
  - 目标窗口未出现
  - 目标控件不存在
  - 脚本超时
  - 验证 oracle 失败
```

## 从用户提示到脚本的转换规则

用户提示通常是不完整但高价值的路线片段。处理方式：

- 先原样写入任务文档或 route reflection。
- 把自然语言步骤拆成 UI 状态转换。
- 对每个动作选择最窄、可观测的实现方式：优先 UIA/Win32/键盘语义动作，坐标点击只能作为明确记录的 fallback。
- 用一次性探针验证提示是否成立。
- 成立后，把该片段封装为函数或脚本步骤。
- 不成立时，记录是提示过期、软件状态不符、观察方式错误，还是自己执行错了。

不要把“用户说过”当作完成证据。用户提示是路线假设，完成证据必须来自当前软件状态或输出 artifact。

## 何时可以脚本化

满足以下条件才写入稳定脚本：

- 已知入口正确，不是相邻但语义不同的菜单或设置页。
- 前置状态可检测，例如编辑模式、目标项目名、无模拟器/监视模式。
- 目标控件可被稳定定位，或键盘路径有独立观察确认。
- 动作后有明确 oracle，例如窗口关闭、字段值回读、树节点出现、编译 OK。
- 错误能被结构化报告，而不是静默继续。

如果只满足部分条件，写 probe 脚本或 breakthrough 脚本，不写 stable 脚本。

## 何时停止推广

遇到以下情况，必须停止扩大声明范围：

- 自动化依赖手动安装、注册、授权或设备连接，但没有干净环境复现。
- 同一路线错误族出现第三次。
- 脚本只在当前脏窗口状态成功。
- 验证依赖截图肉眼判断，缺少机器可读 artifact。
- 编译/保存/应用结果没有同轮证据。
- 用户指出当前路线不符合业务操作逻辑。

停止推广不是停止工作。应当把已验证部分做成脚本，把未验证能力显式拒绝或标注为突破任务。

## 稳定脚本验收清单

- 可在干净 disposable 项目上从已知前置状态运行。
- 输出 JSON 包含 `ok`、`route`、输入参数、关键 UI 观察、证据路径。
- 失败时有稳定错误码。
- 不覆盖无关脚本、不修改用户真实项目，除非用户明确指定。
- 使用独立 oracle 验证结果；工业软件优先用编译/转换/导出/回读结果。
- 同一脚本至少经过一次完整端到端运行；高风险路线需要多次或多设备验证。
- 文档写清楚支持范围和不支持范围。

## 常见错误纠正

- **把相似入口当成正确入口**：例如调试通信设置与项目单元配置是不同语义。先验证最终业务对象是否变化。
- **一步成功后夸大为全流程稳定**：只把成功片段脚本化，继续突破后续片段。
- **模式错误**：如进入模拟器/监视模式后继续编辑。脚本必须检测并恢复或拒绝。
- **错误 oracle**：快捷键不是编译证明，编译结果文本才是证明。
- **把失败归因于时间不够**：增加 timeout 前先证明目标窗口最终确实会出现。
- **重复错误路线**：用户纠正后，立即把旧路线标为废弃，禁止自动回退。

## 输出格式

完成一次 UI 学习自动化任务时，最终汇报应包含：

- 已验证稳定路线。
- 已脚本化的入口和参数。
- 证据路径。
- 明确不支持或未稳定的边界。
- 已提交的 commit。
- 剩余突破任务，如果还有。
