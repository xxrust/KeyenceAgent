# UI 工作流知识提炼成 Skill

## 核心目标

把一次艰难 UI 操作学习中真正沉淀下来的知识，整理成以后可执行、可验证、可避免重犯错误的 skill。这个 skill 关注“如何写成 skill”，不指导 agent 如何继续探索界面。

最终产物应该回答：

- 什么场景会触发这个 skill。
- 哪些 UI 路径和脚本已经验证。
- 哪些用户纠正必须长期保留。
- 哪些错误路线必须禁止。
- 哪些能力还不能声明稳定。
- 后续 agent 应该怎样使用脚本、证据和边界完成任务。

## 输入材料

写 skill 前先收集当前任务的真实材料：

- 用户的关键纠正和提示。
- 已验证成功的脚本、命令和参数。
- 失败过的入口、快捷键、菜单、模式和误判。
- 同轮证据：JSON、截图、导出文件、编译结果、日志。
- 知识库或帮助文档中被证明有用的概念。
- 已提交的代码改动和文档改动。
- 明确未完成或不稳定的边界。

不要把聊天里的印象当事实。只把能在文件、脚本、日志或当前软件行为中找到证据的内容写入 skill。

## 提炼步骤

1. **抽取稳定操作**
   - 只保留已经在真实软件中验证过的入口、控件、快捷键、参数和结果。
   - 把“看起来应该可行”的内容放到未验证边界，不写成操作指令。

2. **抽取用户纠正**
   - 用户纠正通常指出模型最容易重复犯的错，要写成硬规则。
   - 示例类型：入口语义错误、模式错误、快捷键错误、拖拽与回车混淆、编译 oracle 错误。

3. **抽取失败教训**
   - 失败教训要写成可执行禁令或检查项，不写成泛泛总结。
   - 弱写法：注意不要点错。
   - 强写法：不要使用“通信设置”入口配置项目 EtherNet/IP；该入口只用于 PC 到 PLC 调试通信。

4. **抽取脚本接口**
   - 写清脚本名称、必需参数、默认输出目录、成功 artifact 和失败错误码。
   - 如果脚本只支持已注册设备或特定型号，必须写清楚。

5. **抽取验证标准**
   - 每个“稳定”声明必须有独立验证方式。
   - 工业软件优先使用编译、转换、导出、回读、保存后重开等 artifact。
   - 截图可以辅助说明，但不能单独作为稳定证据。

6. **抽取边界**
   - 明确哪些能力没有完成，后续 agent 不得自动推广。
   - 未稳定能力应当在脚本中拒绝，或者在 skill 中标为突破任务。

## Skill 写法

### Frontmatter

`description` 要同时说明能力和触发场景。它决定 Codex 什么时候加载 skill，因此必须具体。

写法：

```yaml
---
name: <skill-name>
description: 面向某软件/某类 UI 任务的稳定操作指导。适用于 Codex 需要创建项目、配置模块、运行脚本、验证编译结果、避免已知错误路线、或复用已验证 UI 自动化流程的任务。
---
```

### 正文结构

推荐结构：

```markdown
# <Skill 标题>

## 适用范围

## 已验证能力

## 标准工作流

## 脚本接口

## 验证标准

## 禁止路线

## 未稳定边界

## 失败处理
```

保持正文面向下一次执行。不要写“我们当时怎么探索”，要写“以后遇到这个任务应该怎么做”。

## 内容取舍

写入 skill：

- 可复用的操作路径。
- 稳定脚本和参数。
- 必须遵守的前置状态。
- 已证伪的错误路线。
- 验证 artifact 的要求。
- 明确错误码和停止条件。

不要写入 skill：

- 长篇过程回忆。
- 没有证据的猜测。
- 一次性实验目录里无复用价值的细节。
- 用户情绪、争执或对话背景。
- 与下一次执行无关的失败堆栈。

## 从探索记录到规则的转换

把原始材料转换成规则时，用下面的格式：

```text
观察/纠正:
  用户指出 SV3 不需要拖拽，找到后按 Enter 即可。

可复用规则:
  EtherCAT SV3 添加设备时，展开 KEYENCE CORPORATION -> Servo Drives，选中 SV3 叶子项后按 Enter。不要使用拖拽作为该路线。

验证要求:
  脚本结果 JSON 必须记录 selected_leaf=SV3，并通过 Ctrl+F9 转换结果。
```

再例如：

```text
观察/失败:
  自动导入 ESI 文件没有完成同轮稳定验证。

可复用规则:
  skill 不声明自动 ESI 注册为稳定能力。

脚本边界:
  参数可以保留，但必须返回明确错误码，提示先手动注册或继续突破。
```

## 稳定声明标准

只有同时满足下面条件，才能在 skill 中写“稳定”：

- 至少一次完整真实软件运行成功。
- 成功不是来自手工补救后的残留窗口状态。
- 有同轮 artifact 证明结果。
- 脚本或流程能从清晰前置状态开始。
- 失败时不会静默污染用户项目或覆盖无关脚本。
- 已知边界写在 skill 中。

如果证据只证明局部步骤，把它写成“已验证片段”；如果只证明思路，把它写成“突破方向”。

## 检查清单

提交 skill 前检查：

- `SKILL.md` frontmatter 有 `name` 和中文 `description`。
- 没有把探索过程当成执行指导。
- 没有把未验证能力写成稳定能力。
- 用户明确纠正已经转成规则或禁令。
- 证据路径或验证方式足够下一次 agent 复查。
- 脚本接口、参数、输出和错误码写清楚。
- 不创建 README、安装指南、过程纪要等无关文件。
- 使用 skill 校验脚本验证结构。

## 最终汇报

完成后向用户报告：

- 新增或修改的 skill 路径。
- 写入的核心规则。
- 明确没有写入或没有声明稳定的边界。
- 校验结果。
- commit。
